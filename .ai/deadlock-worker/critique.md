# Plan Critique

## Summary

The plan correctly identifies the root cause of the deadlock and the proposed fix (lifting lease acquisition to wrap the full worker loop) is directionally sound. However, the plan contains several significant gaps: it mischaracterizes the initiating thread's behavior under the fix, glosses over a residual race with `setOwningThreadPool`/dormant-lease interaction, proposes applying the same change to `DefaultConditionalExecutionQueue` without adequate justification, and underestimates the difficulty of writing a meaningful unit test with a fake `WorkerLeaseService`.

---

## Findings

### Completeness

- [ ] C1: **The initiating thread's `runOperations()` call bypasses `setOwningThreadPool`.** In `waitForCompletion()` (line 141 of `DefaultBuildOperationQueue.java`), the initiating thread calls `new WorkerRunnable(null, parent).runOperations()` directly — it never goes through `WorkerRunnable.run()`, so `OWNING_WORKER_THREAD_POOL` is never set for that thread. The plan's assertion that "the initiator drains what it can" is correct mechanically, but under the proposed fix this `runOperations()` call reaches `runAsWorkerThread`. Since the initiating thread already holds a worker lease, `runAsWorkerThread`'s "already a worker" short-circuit fires (line 125–128 of `DefaultWorkerLeaseService.java`) and the body runs immediately. This is actually fine for correctness in the normal path — but it means the initiating thread's worker loop is **not** associated with any `WorkerThreadPool` via `OWNING_WORKER_THREAD_POOL`. If the initiating thread subsequently calls `workerLeases.blocking(...)` inside `waitForWorkToComplete()`, `withoutLocksBlocking` (line 376 of `DefaultWorkerLeaseService.java`) reads `OWNING_WORKER_THREAD_POOL.get()` to call `notifyBlockingWorkStarting()` — and gets `null` because the initiating thread never called `setOwningThreadPool`. The plan does not acknowledge this gap, so it is unclear whether the compensation mechanism works correctly for the initiating thread in any case where `pendingOperations > 0` after the initiating thread's drain pass.

- [ ] C2: **The `DefaultConditionalExecutionQueue` fix location is wrong.** The plan says to apply "the same change" to `DefaultConditionalExecutionQueue.ExecutionRunner.runBatch`. In that class `ExecutionRunner.run()` (line 120–130) properly calls `setOwningThreadPool` before the work loop, and `runBatch` (line 162) contains the `runAsWorkerThread` call that mirrors the `DefaultBuildOperationQueue` bug. However, unlike `DefaultBuildOperationQueue`, there is no initiating-thread `runOperations()` shortcut in `DefaultConditionalExecutionQueue` — callers use `submit()` and the queue is fully worker-driven. The plan does not analyse whether moving `runAsWorkerThread` outside the inner loop in `DefaultConditionalExecutionQueue` would break the `isExtraWorker()` / early-exit logic. Inside the current `runBatch`, the worker checks `isExtraWorker()` and calls `invalidateIfNeeded()` before returning from within the `runAsWorkerThread` lambda. If the `runAsWorkerThread` call is lifted outside `waitForNextOperation`, the worker would hold a lease while blocked in `workAvailable.await()`. This is a behavior change in a long-lived queue (with 2-second keep-alive) that has no analog to the `DefaultBuildOperationQueue` scenario.

- [ ] C3: **The `doRunBatch` cancelled-operation discard path is unanalyzed.** `doRunBatch` (line 322–327) discards a polled operation (returning `operationCount++`) when `queueState == Cancelled`. Under the proposed fix, if `cancel()` is called while a worker is inside `runAsWorkerThread` (before it enters `doRunBatch`), the polled operation was already removed from the queue but the worker has not yet decremented `pendingOperations` via `completeOperations`. The plan states the fix ensures "operations remain in the queue," but does not address what happens to operations that were returned from `waitForNextOperation()` (now renamed `pollWork` in the proposed `pollAndRunOperations`) before the cancellation arrives.

### Backward Compatibility

No issues found. The changes are to internal classes with no public API surface. No `@Incubating`, `@since`, or `accepted-public-api-changes.json` implications.

### Testing Strategy

- [ ] C4: **The unit test design is not feasible as described.** The plan proposes using a "mock `WorkerLeaseService` that causes `runAsWorkerThread` to block indefinitely from spawned worker threads (but returns immediately for the caller thread that already holds a lease)." The `DefaultWorkerLeaseService.runAsWorkerThread` already provides the "already a worker" short-circuit based on whether the thread currently holds a lock in `workerLeaseLockRegistry`. A fake that replicates this behavior correctly requires knowing which thread is calling it. A Spock/Groovy mock won't provide this naturally — the test will likely be brittle or require a real `DefaultWorkerLeaseService` with a saturated lease count. The existing `DefaultBuildOperationQueueTest` already uses a real `DefaultWorkerLeaseService` with controlled worker counts and latches (`SynchronizedBuildOperation`). The plan should use that pattern — start a queue with `maxWorkers=N`, saturate all leases with blocking operations from the outside, then submit work items, and verify they complete without hanging — rather than a fake `WorkerLeaseService`.

- [ ] C5: **The integration test approach is impractical and may be impossible to reliably reproduce.** The plan proposes a composite build with N >= 10 included builds, each triggering shaded-jar generation with `--max-workers=3`. The shaded-jar generation path goes through `RuntimeShadedJarFactory`, which caches aggressively — once the JAR exists, subsequent builds hit the cache and never exercise the code path. The test would need to arrange for the cache to be cold on every run, prevent parallel initialization from completing before the race window, and do so in a way that is stable across CI machines with varying scheduler behavior. The plan does not address any of this. The suggested location `platforms/core-runtime/base-services/src/integTest/` is also wrong: `DefaultBuildOperationQueue` lives in `:core` (`:subprojects:core`), and integration tests that exercise `runAll` with worker-lease contention already exist in `subprojects/core/src/test/groovy/org/gradle/internal/operations/MaxWorkersTest.groovy`. A more targeted integration test using `MaxWorkersTest`-style structure would be both simpler and more reliable.

- [ ] C6: **No regression coverage for the `isExtraWorker` / worker-count-reduction path.** Under the proposed fix, a worker that holds a lease and is waiting in `workAvailable.await()` is counted in `workerCount` but also holds a lease. If `isExtraWorker()` fires mid-loop, the worker calls `invalidateIfNeeded()` (decrementing `workerCount`) before the `runAsWorkerThread` lambda returns. This sequence is not tested by any existing test, and the plan does not add one.

### Performance

- [ ] C7: **The "holding the lease while idling" tradeoff is understated.** The plan acknowledges the tradeoff in one paragraph but dismisses it as minor. The argument that "the common pattern is `runAll` where work is added synchronously before `waitForCompletion`" is plausible for `DefaultBuildOperationQueue`, but the footnote that "a follow-up can refine `waitForNextOperation` to release the lease via `workerLeases.blocking()`" is exactly the behavior that caused the original deadlock. The plan should be explicit that any such follow-up must include the corresponding `setOwningThreadPool` setup for the initiating thread (see C1), and should defer that refinement rather than suggesting it without caveats.

- [ ] C8: **Impact on `DefaultConditionalExecutionQueue` under lease pressure is not analyzed.** Unlike `DefaultBuildOperationQueue`, which is a short-lived object tied to a single `runAll` call, `DefaultConditionalExecutionQueue` is a long-lived service with a 2-second keep-alive. Workers in that queue currently release their lease between work items (the `runAsWorkerThread` wraps only `runBatch`). After the proposed change, a worker waiting for more work will hold its lease for up to 2 seconds of idle time. Under lease contention (e.g., while task workers are waiting), this means `max-workers` leases could be held by idle `DefaultConditionalExecutionQueue` threads. This is a materially different and potentially harmful behavior change that the plan waves away without measurement.

### Cross-cutting Concerns

- [ ] C9: **`setOwningThreadPool` interaction is not fully composed.** The dormant-lease mechanism in `withoutLocksBlocking` (line 375–399 of `DefaultWorkerLeaseService.java`) reads `OWNING_WORKER_THREAD_POOL.get()` to call `notifyBlockingWorkStarting`. Under the proposed fix, worker threads in `DefaultBuildOperationQueue` still call `setOwningThreadPool` in `WorkerRunnable.run()`. However, the initiating thread does not (see C1). This means: if the initiating thread's `pollAndRunOperations()` (proposed) encounters an operation that internally calls `workerLeases.blocking(...)` (e.g., a nested `runAll`), the `notifyBlockingWorkStarting` compensation is silently skipped for the initiating thread. This gap pre-existed (before PR #36897, the initiating thread didn't have `setOwningThreadPool` set either), but the plan's diff sketch changes the initiating thread's code path and should explicitly address this.

- [ ] C10: **Configuration cache compatibility is assumed but not verified.** The plan says to run `configCacheIntegTest` but does not identify any configuration-cache-specific concern. `DefaultBuildOperationQueue` is used at configuration time (e.g., for script compilation and plugin resolution operations), so verifying CC compatibility is non-trivial. The plan should at minimum identify which callers of `runAll` with `requiresWorkerLease=true` execute at configuration time.

- [ ] C11: **No Windows / OS-specific concerns identified**, and none appear to exist for this change. No issues found.

### Risks and Assumptions

- [ ] C12: **The plan's core claim ("operations remain in the queue if lease unavailable") is only true if `waitForNextOperation` is never called before the lease is held.** In the proposed diff sketch, `pollAndRunOperations` is called inside the `runAsWorkerThread` lambda. This is correct — `pollWork` is only called after the lease is acquired. But the plan's description in "Why this works" says "the queue initiator ... can pick it up because it was already removed from the queue." This statement is wrong in the base case: if workers block waiting for a lease *before* calling `pollWork`, then no operation has been removed from the queue yet, so the initiating thread can drain all operations. The plan gets the mechanics right in the code but gets the explanation wrong in the prose, which suggests the author may not have fully internalized the implication. Specifically: the plan says the fix makes the queue work by ensuring "every polled op was either completed by the initiator or never polled by a stuck worker." After the fix there *are no* polled-but-unstuck-ops — workers either hold a lease and proceed, or block before polling. The initiating thread drains everything. The explanation should be corrected, as an incorrect rationale risks introducing the bug again in a future refactor.

- [ ] C13: **Rollback strategy is not mentioned.** Given that PR #36897 itself was a targeted structural change, and the proposed fix further restructures the lease-acquisition point, a rollback plan (e.g., a feature flag or a partial revert of #36897) should be identified in case the fix causes unexpected lease-starvation regressions in CI.

- [ ] C14: **The fix for `DefaultConditionalExecutionQueue` is scope creep without a confirmed bug.** The plan explicitly states "there is no confirmed deadlock report for that queue" but proposes applying the same change "for symmetry." Given the behavioral differences (long-lived queue, no initiating-thread drain, lease-hold-during-idle tradeoff), this change should either be backed by analysis showing it is necessary or deferred to a separate issue.

## Recommended Changes

1. **Remove the `DefaultConditionalExecutionQueue` change from this PR** (addresses C2, C8, C14). It has no confirmed bug report, has meaningfully different semantics from `DefaultBuildOperationQueue`, and the lease-hold-while-idle concern is worse there. File a separate investigation issue if symmetry is genuinely desired.

2. **Clarify or add `setOwningThreadPool` for the initiating thread** (addresses C1, C9). Either document explicitly that the initiating thread's `runOperations()` call intentionally operates without `OWNING_WORKER_THREAD_POOL` being set (because it already holds a lease and the "already a worker" short-circuit means `withoutLocksBlocking` is never triggered by that call path), or add `setOwningThreadPool(this)` / `setOwningThreadPool(null)` around the initiating thread's `runOperations()` call in `waitForCompletion()`.

3. **Replace the fake-`WorkerLeaseService` unit test design with a real-service, coordinated-latches approach** (addresses C4). Model on the existing `SynchronizedBuildOperation` pattern in `DefaultBuildOperationQueueTest` or the `MaxWorkersTest` async-block pattern. Specifically: create a queue with `maxWorkers=2`, start 2 external threads that hold leases and block on a latch, submit 3 work items, call `waitForCompletion()` from a third thread that also holds a lease, then release the external threads' latches and assert all 3 items complete without timeout.

4. **Replace or heavily constrain the composite-build integration test** (addresses C5). Instead of a full composite-build + shaded-jar repro (which is fragile and environment-dependent), write a `forkingIntegTest` that uses the `MaxWorkersTest` style: a build that explicitly saturates worker leases from one thread while another thread calls `buildOperationExecutor.runAll` with `requiresWorkerLease=true`, and asserts the build completes without hanging (use `@Timeout` on the test). The existing `MaxWorkersTest` is a unit-level concurrent spec — promoting one scenario to a real forking integration test is simpler than the shaded-jar repro.

5. **Add explicit test for the `isExtraWorker` / early-exit path under the new lease-wrapping** (addresses C6). This is a behavioral difference from the current code that has no test coverage.

6. **Correct the prose in "Why this works"** (addresses C12). The claim that "every polled op was either completed by the initiator or never polled by a stuck worker" is confusing. After the fix, stuck workers never poll — so the correct statement is: "stuck workers block before polling, so all operations remain in the queue and are drained by the initiating thread."

7. **Add a rollback strategy to the plan** (addresses C13). Minimally note that if lease-starvation regressions are observed in CI, the fix can be reverted and the fallback is a partial revert of PR #36897 (removing `requiresWorkerLease` from `DefaultBuildOperationQueue`).
